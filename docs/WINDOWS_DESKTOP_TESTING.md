# Running the Flutter desktop app on Windows (against `stack-connect-core`)

> **Audience:** an agent/LLM (or developer) on a **Windows** machine tasked with
> building and running the `stack_desktop` Flutter app against this Rust core.
> This document is self-contained — it assumes no prior context about the repos.

## 0. Architecture in one minute

- **`stack-connect-core`** (this repo, Rust): the shared core. Exposes a
  [`flutter_rust_bridge`](https://cjycode.com/flutter_rust_bridge/) (FRB) facade
  (`crates/stack_core/src/frb_api.rs`) behind the cargo feature `frb` (OFF by
  default). The generated Dart binding is a standalone package at
  `bindings/dart/stack_core_rust` (mirrors how the Swift binding lives under
  `bindings/swift`).
- **`stack-connect`** (sibling repo, Flutter): the apps. `flutter/` is a Dart
  pub **workspace** containing `apps/stack_mobile` (Android, Material 3),
  `apps/stack_desktop` (**Windows**, fluent_ui), and the package
  `packages/stack_core_dart` (hand-written Riverpod controllers + host stores).
  `stack_core_dart` consumes the binding from this core repo via a `path`
  dependency on `bindings/dart/stack_core_rust`.

**What is committed vs generated (important):**

| Artifact | State | How to (re)create |
| --- | --- | --- |
| `bindings/dart/stack_core_rust/lib/src/rust/*` (Dart binding) | **gitignored** | `just gen-dart` (§3.1) |
| `crates/stack_core/src/frb_generated.rs` (Rust glue) | tracked, but regenerated by codegen | `just gen-dart` |
| `stack_core.dll` (native lib) | not committed | `just build-desktop` (§3.2) |

So on a **fresh clone the Dart binding and the DLL do not exist** — you must
generate/build them locally before the desktop app can run.

## 1. Prerequisites (install on the Windows machine)

- **Rust** 1.96.0 (the repo pins it via `rust-toolchain.toml`) + the
  **`x86_64-pc-windows-msvc`** target. On a Windows host this is the default host
  triple, so `rustup` already has it; otherwise `rustup target add x86_64-pc-windows-msvc`.
- **Visual Studio Build Tools** with the **“Desktop development with C++”**
  workload (MSVC toolchain + Windows SDK + CMake). Required both for the Rust
  MSVC target and for Flutter Windows desktop.
- **Flutter** (stable) with Windows desktop enabled: `flutter config --enable-windows-desktop`.
  Verify with `flutter doctor` (Windows toolchain must be green).
- **`flutter_rust_bridge_codegen` 2.12.0** — must match the `flutter_rust_bridge`
  runtime crate (`crates/stack_core/Cargo.toml`) and the Dart dep
  (`bindings/dart/stack_core_rust/pubspec.yaml`), both pinned to `=2.12.0`.
  Install: `cargo install flutter_rust_bridge_codegen --version 2.12.0`.
- **`just`** (optional but recommended): `cargo install just`. The recipes wrap
  `bash` scripts, so to use `just gen-dart` / `just build-desktop` you also need
  **Git Bash / MSYS2 / WSL**. §3 gives raw PowerShell equivalents if you prefer
  to skip bash.

## 2. Repo layout

Clone both repos **as siblings** (the binding path dependency and the build
scripts assume `stack-connect-core` and `stack-connect` share a parent dir):

```
<parent>/
  stack-connect-core/   # branch: master
  stack-connect/        # branch: experiment/flutter   (the Flutter app lives in flutter/)
```

```powershell
cd <parent>
git clone <core-remote>    stack-connect-core
git clone <flutter-remote> stack-connect
git -C stack-connect-core checkout master
git -C stack-connect       checkout experiment/flutter
```

The Flutter package’s path dep is relative:
`stack_core_rust: { path: ../../../../stack-connect-core/bindings/dart/stack_core_rust }`
— the sibling layout makes it resolve.

## 3. Build steps (run in order, from `stack-connect-core/`)

### 3.1 Generate the Dart binding

This regenerates the gitignored `lib/src/rust/*` Dart files **and** the core’s
`frb_generated.rs`, with the correct FFI symbol prefix (`frbgen_stack_core_rust_*`).

**With Git Bash / `just`:**
```bash
just gen-dart
```

**Raw PowerShell equivalent** (what `build/gen-dart.sh` does):
```powershell
cd bindings\dart\stack_core_rust
flutter pub get
flutter_rust_bridge_codegen generate
# Workspace target-dir fix (gen-dart.sh does this via sed): the FRB default
# loader bakes ioDirectory as `<crate>/target/release`, but this is a cargo
# WORKSPACE so the real output is the workspace-root `target/release`.
(Get-Content lib\src\rust\frb_generated.dart) `
  -replace 'crates/stack_core/target/release','target/release' `
  | Set-Content lib\src\rust\frb_generated.dart
dart run build_runner build --delete-conflicting-outputs
cd ..\..\..
```

### 3.2 Build the Windows native library (`stack_core.dll`)

**With Git Bash / `just`:**
```bash
just build-desktop            # defaults to x86_64-pc-windows-msvc
# or: ./build/build-desktop.sh windows
```

**Raw PowerShell equivalent:**
```powershell
rustup target add x86_64-pc-windows-msvc   # no-op if already present
cargo build --release -p stack_core --features frb --target x86_64-pc-windows-msvc
# Produces: target\x86_64-pc-windows-msvc\release\stack_core.dll
```

> The FRB loader finds the library by **stem** `stack_core`, i.e. it looks for
> `stack_core.dll` **next to the running executable**. The `ioDirectory` baked
> into the binding is only a dev fallback and (because the binding lives in a
> different repo than the app) will **not** reliably reach the core’s `target/`
> dir at runtime. So the DLL must be bundled next to the exe — see §4.

### 3.3 Resolve Flutter deps

```powershell
cd ..\stack-connect\flutter
flutter pub get
```

## 4. Bundle `stack_core.dll` next to the executable

The FRB loader finds the lib by stem `stack_core`, i.e. `stack_core.dll` must sit
next to `runner.exe`. This is the one runtime requirement.

### Option A — CMake install step (ALREADY WIRED — just verify)

The app-level **`stack-connect/flutter/apps/stack_desktop/windows/CMakeLists.txt`**
already has an `install(FILES "${STACK_CORE_DLL}" DESTINATION "${INSTALL_BUNDLE_LIB_DIR}" …)`
block (added in the `experiment/flutter` branch) that copies the Rust DLL next to
the executable. It defaults `STACK_CORE_DLL` to the **sibling-repo** path
`…/stack-connect-core/target/x86_64-pc-windows-msvc/release/stack_core.dll` and
emits a configure-time **WARNING** if the DLL is missing.

**This change has NOT been validated on Windows** (it was authored on macOS). On
the Windows machine, your job is to:
1. Build the DLL first (§3.2) so the default path exists.
2. If the repos are **not** checked out as siblings, pass the absolute path:
   `flutter run -d windows --dart-define=…` won’t forward it — instead build with
   an explicit CMake var, e.g. add to the app’s `windows` build or run
   `cmake -DSTACK_CORE_DLL=<abs path> …` if invoking CMake directly. Simplest:
   keep the sibling layout so the default resolves.
3. Confirm after a build that `stack_core.dll` landed in
   `build\windows\x64\runner\{Debug,Release}\`. If the configure step printed the
   “stack_core.dll not found” warning, the path is wrong — fix `STACK_CORE_DLL`.

### Option B — quick manual test (bypass CMake entirely)

Build once (`flutter build windows` or `flutter run -d windows` — it will start
but fail to load the lib), then copy the DLL next to the produced exe:

```powershell
copy ..\..\stack-connect-core\target\x86_64-pc-windows-msvc\release\stack_core.dll `
     build\windows\x64\runner\Release\
# (use ...\runner\Debug\ for a debug run)
```

Or set `DEST=` when building so the script copies it for you:
```bash
DEST="../stack-connect/flutter/apps/stack_desktop/build/windows/x64/runner/Release" \
  ./build/build-desktop.sh windows
```
(The `build/...` dir only exists after a first Flutter build, so build the app
once before relying on `DEST`.)

## 5. Run & test

```powershell
cd stack-connect\flutter\apps\stack_desktop
flutter run -d windows
```

`apps/stack_desktop/lib/bootstrap.dart` calls `await RustLib.init();` (no explicit
library path) — so it relies on the stem-based lookup, i.e. §4 must be done.

**Tests:**
- Controller/widget tests that **fake `CoreGateway`** need no native lib:
  `flutter test apps/stack_desktop` and the controller tests in
  `packages/stack_core_dart` run without the DLL.
- The one integration test `packages/stack_core_dart/test/stack_core_dart_test.dart`
  hardcodes a **macOS absolute `.dylib` path** and will fail on Windows as-is.
  To exercise it on Windows, change `_hostDylibPath` to the built DLL, e.g.
  `r'...\stack-connect-core\target\x86_64-pc-windows-msvc\release\stack_core.dll'`
  (or build the host debug lib and point at it). This file lives in the
  `stack-connect` repo.

## 6. Verification checklist

- [ ] `flutter doctor` — Windows toolchain green.
- [ ] `cargo build --features frb --target x86_64-pc-windows-msvc` succeeds → `stack_core.dll` exists.
- [ ] `bindings/dart/stack_core_rust/lib/src/rust/` populated after `gen-dart`.
- [ ] `flutter pub get` clean at the workspace root.
- [ ] `flutter analyze` clean for `packages/stack_core_dart` and `apps/stack_desktop`.
- [ ] DLL bundled next to `runner.exe` (§4).
- [ ] `flutter run -d windows` launches and reaches the accounts screen.

## 7. Troubleshooting

- **`Failed to load dynamic library 'stack_core.dll'` / lib not found at runtime**
  → §4 not done, or the DLL isn’t next to the exe. Confirm `stack_core.dll` sits
  in `build\windows\x64\runner\{Debug,Release}\`.
- **`Content hash … different from Rust side`** → the Dart binding and the native
  lib are out of sync. Always rebuild the DLL (§3.2) **after** running codegen
  (§3.1). They must come from the same generation.
- **Undefined/mismatched `frbgen_*` symbols** → the binding package name drives the
  FRB symbol prefix (here `frbgen_stack_core_rust_*`). Don’t rename the package;
  regenerate both sides together via `gen-dart`.
- **`flutter_rust_bridge_codegen` version mismatch** → it must be exactly `2.12.0`
  to match the runtime crate and the Dart dep.
- **`just`/scripts fail on `sed`, `du`, `$HOME`** → the `build/*.sh` scripts are
  bash; run them under Git Bash/WSL, or use the PowerShell equivalents in §3.
- **MSVC link errors building the crate** → ensure the “Desktop development with
  C++” workload is installed and you’re in a shell that sees the MSVC toolchain.

## 8. Key files

| File | Role |
| --- | --- |
| `crates/stack_core/src/frb_api.rs` | the FRB facade (all 15 capabilities) |
| `bindings/dart/stack_core_rust/flutter_rust_bridge.yaml` | FRB codegen config |
| `bindings/dart/stack_core_rust/lib/stack_core_rust.dart` | barrel re-exporting the binding |
| `build/gen-dart.sh` | regenerate the Dart binding (+ ioDirectory workspace fix) |
| `build/build-desktop.sh` | build the desktop cdylib (`stack_core.dll`) |
| `justfile` | `just gen-dart`, `just build-desktop`, … |
| `stack-connect/flutter/apps/stack_desktop/windows/**/CMakeLists.txt` | where §4 bundling goes |
| `stack-connect/flutter/apps/stack_desktop/lib/bootstrap.dart` | calls `RustLib.init()` |
